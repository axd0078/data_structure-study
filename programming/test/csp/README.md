## 实现方法

### 1. T1 题优先稳住线性扫描和格式控制

`3/1.cpp`、`21/1.cpp`、`22/1.cpp`、`25/1.cpp` 这类题基本都可以在一次顺序扫描里完成。关键不是技巧，而是：

- 读入后是否需要立即输出
- 用 `int` 还是 `long long`
- 首项和后续项之间的空格、换行是否处理正确
- 状态更新顺序是不是符合题意

### 2. 模拟题先把状态画清楚

`1/2.cpp`、`36/1.cpp`、`40/3.cpp` 和 `41/3.cpp` 都属于状态模拟题。写代码前先确定：

- 当前状态有哪些变量
- 每个操作会修改哪些状态
- 是否需要维护多个索引结构
- 输出发生在更新前还是更新后

### 3. 小状态空间优先考虑预处理

`40/2.cpp` 的状态总数只有 `512` 个。遇到这种“状态小、查询多”的题，优先考虑整表预处理，而不是每次查询都重跑完整过程。

### 4. 无法反推的摘要要维护历史表

`38/3.cpp` 的 12 位和 25 位哈希摘要不能直接反推原始代码串，所以必须维护“此前出现过的代码 -> 哈希值”的历史映射。题目重点不是解密，而是按时间顺序查历史。

### 5. 区间覆盖类题目优先想差分

`39/2.cpp` 里每个 `5 x 9` 窗口都会给出一个阈值区间。如果对每个区间逐个枚举阈值，复杂度会偏高；这类“很多区间加一，最后看哪些点被覆盖”的题，优先考虑差分数组。

## 运行方式

在仓库根目录下编译某个文件：

```bash
g++ -std=c++11 programming/test/csp/3/1.cpp -o test && ./test
g++ -std=c++11 programming/test/csp/21/1.cpp -o test && ./test
g++ -std=c++11 programming/test/csp/36/4.cpp -o test && ./test
g++ -std=c++17 programming/test/csp/38/3.cpp -o test && ./test
```

如果当前目录在 `programming/test` 下：

```bash
g++ -std=c++11 csp/22/1.cpp -o test && ./test
g++ -std=c++11 csp/25/1.cpp -o test && ./test
g++ -std=c++11 csp/41/3.cpp -o test && ./test
```

## 注意事项

- `3/1.cpp` 这次改动的重点是去掉首项前导空格，输出格式题要优先自测样例。
- `21/1.cpp` 和类似加权求和题建议默认用 `long long` 兜底。
- `22/1.cpp` 的灰度值范围由 `L` 控制，计数数组大小要和输入范围严格一致。
- `23/1.cpp` 当前实现一次遍历同时计算两类答案，复盘时重点看状态量的含义是否稳定。
- `25/1.cpp` 中编号 `0` 被视为已初始化，这种“天然合法源”要在初始化阶段先标出来。
- `38/3.cpp` 和 `41/3.cpp` 都属于实现细节明显多于算法名本身的题，复盘时更适合按“字段拆分 / 状态维护 / 更新顺序”来记。
