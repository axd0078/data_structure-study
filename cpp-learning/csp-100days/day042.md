# Day 042：嵌套结构与时间枚举

## 学习内容

今天两题都在练“先把规则结构化，再按查询或时间线输出”。第11次 T3《JSON查询》要求把简化 JSON 解析成树或嵌套 map，再按点分路径查询；第12次 T3《Crontab》要求把五个时间字段解析成可匹配集合，再枚举时间点输出命令。

《JSON查询》的关键是递归下降解析。题目中的 JSON 只有对象和字符串两类值，对象是若干键值对，字符串可能含 `\"` 和 `\\` 转义。解析时维护当前位置 `pos`，遇到 `{` 解析对象，遇到 `"` 解析字符串。解析完后，查询 `a.b.c` 就是从根对象沿着 key 逐层走。

《Crontab》的关键是字段展开和时间合法性。分钟、小时、日期、月份、星期字段都可以出现 `*`、逗号、短横线区间以及英文缩写。先把每个字段转成一个布尔集合或有序数组，再枚举开始时间到结束时间之间的日期，对日期、月份、星期做过滤，然后枚举当天可用小时和分钟。最后按时间升序、同一时间按规则输入顺序输出。

这两题都提醒我们：T3 的“模拟”往往不是直接模拟原文，而是把规则先变成程序舒服的数据结构。结构化做得好，后面的判断会短很多。

## 题解

### 第11次 T3 JSON查询

**题意抓手**

给出一个简化 JSON 文档和若干查询。查询用 `.` 分隔路径，若路径不存在输出 `NOTEXIST`；若查到字符串输出 `STRING value`；若查到对象输出 `OBJECT`。

**知识点**

递归下降解析器。用一个 `Node` 表示 JSON 值：如果是字符串，保存 `value`；如果是对象，保存 `map<string, Node>`。解析函数根据当前位置字符决定解析对象还是字符串。

**解法**

先把 JSON 的多行输入拼成一个字符串。解析对象时跳过 `{`，循环读取字符串键、冒号和值，直到遇到 `}`。解析字符串时从开引号后开始扫描，普通字符直接加入结果，遇到反斜杠就把下一个字符作为真实字符加入。查询时把路径按 `.` 拆分，从根节点开始逐层在对象子节点中查找。

**易错点**

- JSON 输入可能跨多行，要先拼接，不要只读一行。
- 字符串中的 `\"` 表示一个双引号，`\\` 表示一个反斜杠。
- 查询过程中如果当前节点已经是字符串，却还要继续往下查，应输出 `NOTEXIST`。
- 键名大小写敏感，不能转小写。

**复杂度**

设 JSON 文本长度为 `L`，查询路径总长度为 `Q`。解析时间复杂度 `O(L)`，查询总复杂度约为 `O(Q log K)`，其中 `K` 是对象内键数量；空间复杂度 `O(L)`。

### 第12次 T3 Crontab

**题意抓手**

给出若干 crontab 规则和一个起止时间区间。每条规则包含分钟、小时、日、月、星期五个字段和命令名。输出所有在区间内触发的命令，按触发时间升序，同一时间按规则输入顺序。

**知识点**

字段展开、日期计算、星期计算、事件排序。`*` 表示全选，`,` 表示多个片段，`a-b` 表示闭区间。月份和星期可出现英文缩写，且不区分大小写。

**解法**

把每条规则的五个字段分别展开为有序数组。枚举从开始日期到结束日期的每一天，先判断月份、日期、星期是否匹配；匹配后枚举规则允许的小时和分钟，构造完整时间，过滤掉区间外时间，再加入事件列表。事件保存 `(time, rule_index, command)`，最后排序输出。

这种做法比逐分钟枚举更稳：时间区间最多几年，按天枚举再组合小时分钟，复杂度主要由真实候选触发点控制。

**易错点**

- 结束时间是不包含的，输出时间必须满足 `start <= time < end`。
- 星期要和题目定义一致，通常 Sunday 为 0。
- 月份和星期缩写不区分大小写，例如 `Jan`、`jan`、`JAN` 等价。
- 日期字段和星期字段在本题中同时约束，二者都要满足。
- 同一触发时间多条命令要保持规则输入顺序。

**复杂度**

设规则数为 `N`，枚举天数为 `D`，每条规则每天最多产生 `H*M` 个候选。时间复杂度可写作 `O(D * N * C + E log E)`，其中 `C` 是每条规则当天候选小时分钟数，`E` 是输出事件数；空间复杂度 `O(N+E)`。
