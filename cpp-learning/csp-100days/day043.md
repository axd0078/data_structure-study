# Day 043：STL 容器组合

## 学习内容

今天练两个“选择器/匹配器”题。第13次 T3《URL映射》把路径规则和真实 URL 做匹配；第14次 T3《元素选择器》把简化 HTML 文档和 CSS 选择器做匹配。共同点是：输入看起来像字符串题，但真正要做的是把字符串拆成结构，再按规则逐段比较。

《URL映射》里，规则和 URL 都可以按 `/` 拆成路径段。普通路径段必须完全相等；`<str>` 匹配非空字符串段；`<int>` 匹配十进制整数段，输出时去掉前导零；`<path>` 匹配剩余路径，只能出现在规则末尾。规则按输入顺序尝试，第一条匹配成功就输出规则名和参数。

《元素选择器》里，文档行通过前导 `..` 表示深度，形成一棵树。选择器可以是标签、id，或者多个选择器组成的后代选择器。判断一个节点是否匹配多段选择器时，从节点本身匹配最后一段，然后沿父节点链向上寻找前面的选择器段。标签匹配不区分大小写，id 匹配区分大小写。

这类题最好先写两个小函数：`split` 负责拆 token，`match_one` 负责判断单个 token 是否匹配。主流程只组合这些函数，不要让大量字符串判断散落在循环里。

## 题解

### 第13次 T3 URL映射

**题意抓手**

给出若干 URL 规则和若干请求路径。每条请求按规则输入顺序匹配，成功后输出规则名和占位符提取出的参数；没有规则匹配输出 `404`。

**知识点**

路径分段和模式匹配。`<int>`、`<str>`、`<path>` 是三种特殊 token。因为规则数量和路径长度都不大，逐条规则模拟匹配即可。

**解法**

预处理每条规则，把路径按 `/` 拆成段，并保存规则名。处理请求时，同样拆成段。按规则顺序比较：

- 普通段必须完全一致。
- `<str>` 匹配一个非空段，参数原样保存。
- `<int>` 匹配全是数字的段，参数转整数后再转字符串，去掉前导零。
- `<path>` 匹配剩余所有段，用 `/` 连接成一个参数，并结束匹配。

如果规则段和请求段都刚好消耗完，说明匹配成功。

**易错点**

- 规则按输入顺序匹配，不能为了方便排序。
- `/static` 和 `/static/` 不等价，尾部斜杠会产生不同路径形态。
- `<int>` 的参数输出要去掉前导零，但 `0` 仍输出 `0`。
- `<path>` 只能作为最后一个规则段，匹配时要吃掉剩余 URL。

**复杂度**

设规则数为 `N`，请求数为 `M`，单条路径最大段数为 `S`。时间复杂度 `O(M*N*S)`，空间复杂度 `O(N*S)`。

### 第14次 T3 元素选择器

**题意抓手**

输入一个用缩进表示层级的简化 HTML 文档，以及若干选择器。输出每个选择器匹配到的元素数量和这些元素的行号。

**知识点**

父指针树、字符串匹配、后代选择器。每个元素只需要保存标签、可选 id、父节点编号和原始行号。查询时逐个元素判断是否匹配。

**解法**

读文档时统计行首 `..` 的个数作为深度。维护 `last_at_depth[depth]`，当前节点的父亲就是 `last_at_depth[depth-1]`。标签统一转小写保存，id 原样保存。

单段选择器如果以 `#` 开头，就匹配 id；否则按小写标签匹配。多段选择器如 `a b c` 表示后代关系：候选节点先匹配 `c`，然后沿父节点链向上找能匹配 `b` 的祖先，再继续找 `a`。所有段都找到才算匹配。

**易错点**

- 标签大小写不敏感，但 id 大小写敏感。
- 后代选择器不要求父子相邻，中间可以隔多层。
- 输出的是原文行号，从 1 开始。
- 文档行可能没有 id，也可能有 `tag #id` 两个 token。

**复杂度**

设文档元素数为 `N`，查询数为 `Q`，选择器最大段数为 `K`，树高为 `H`。逐点匹配时间复杂度 `O(Q*N*K*H)`，本题规模下足够；空间复杂度 `O(N)`。
